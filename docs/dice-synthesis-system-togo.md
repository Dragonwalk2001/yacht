# 骰子合成与远征系统设计说明（TOGO）

本文定义下一版目标行为，聚焦三项改动：

1. 重做远征等待阶段（可离开窗口、各桌独立）
2. 调整合成前置条件（需达到“该桌最大骰量 + 1”）
3. 重做合成算法（随机打乱后逐位取最大）与稀有度规则

当前相关实现文件（落地时预计涉及）：

- `scripts/ui_expedition.gd`
- `scripts/game_state.gd`
- `scripts/die_definition.gd`
- `scripts/ui_controller.gd`（状态展示与提示）

## 1. 远征等待阶段重做

## 1.1 目标行为

- 远征开始后，玩家可以关闭远征窗口并继续进行其他操作。
- 各骰桌远征完全独立，可同时存在多桌处于远征中。
- 某桌远征结束后，仅该桌进入“待确认结果”状态，不阻塞其他桌操作。

## 1.2 核心约束

- 不再使用“全局唯一进行中远征”限制。
- 不再使用“结果未确认时锁死所有桌远征入口”限制。
- 远征窗口只是查看/操作面板，不是流程锁本体；流程状态必须由每桌数据驱动。

## 1.3 每桌状态模型（建议）

每张桌子独立维护远征状态：

- `idle`：空闲，可发起远征
- `running`：计时中
- `ready_to_confirm`：计时结束，待玩家确认结果

并为每桌维护：

- `pending_kind`（获得/删骰/合成）
- `pending_candidates`（候选数据快照）
- `started_at` / `finish_at` 或等价计时信息

说明：

- 候选应在“进入待确认结果时”生成并固定，避免玩家反复开关窗口导致候选变化。

## 1.4 UI 交互要求

- 远征窗口可随时关闭；关闭不取消流程。
- 再次打开任一桌远征窗口时，应恢复该桌当前状态：
  - `running`：显示剩余时间与进行中状态
  - `ready_to_confirm`：直接展示既有候选，允许确认
- 支持在 A 桌等待期间，去 B 桌发起远征或进行常规掷骰/结算。

## 2. 合成前置条件调整

## 2.1 新规则

合成远征在“开始前”和“确认前”都必须满足：

`当前该桌骰子数 >= 该桌最大骰子量 + 1`

其中“该桌最大骰子量”指当前科技条件下该桌可上场骰子上限。

## 2.2 设计意图

- 合成是“压缩骰子数量换质量”，只在“超出编制”时允许触发。
- 避免玩家在低骰量阶段频繁合成，导致节奏前置失衡。

## 2.3 校验时机（必须双校验）

1. 点击“开始远征”时校验一次（快速反馈）
2. 点击“确认结果”时再校验一次（防止等待期间状态变化导致违规执行）

若确认时不满足条件，应拒绝结算并给出明确提示，不消耗本次结果选择（可按产品决定是否保留候选）。

## 3. 新合成算法

## 3.1 面值融合规则（替换旧规则）

给定两个骰子 `A`、`B`：

1. 分别取出 `A.faces[6]`、`B.faces[6]`
2. 各自进行一次随机打乱（shuffle）
3. 对打乱后的两个数组逐位取最大，得到 `M[6]`
4. 将 `M` 排序（升序）作为新骰子的最终六面

即：

`merged_faces = sort_asc(max(shuffle(A.faces), shuffle(B.faces)))`

说明：

- 新规则增加“配对随机性”，降低固定面位带来的确定性收益。
- 最终排序保证展示与后续认知一致（面集表达稳定）。

## 3.2 稀有度规则

- 若两颗骰子稀有度不同：取较高者
- 若两颗骰子稀有度相同：新骰子稀有度 = 原稀有度 + 1

可表达为：

`rarity_out = max(r1, r2) + (1 if r1 == r2 else 0)`

## 3.3 稀有度上限（待定项）

需要明确是否存在全局上限 `RARITY_MAX`：

- 若有上限：结果做 `clamp`，避免无限增长
- 若无上限：需同步评估后期数值爆炸风险（尤其与远征获取来源叠加）

建议默认引入上限并在配置中集中管理。

## 3.4 buff 合并策略

本次需求未指定 buff 规则变更，默认保持现有策略不变（拼接/继承）。

如后续需要，可单独补充为可配置策略。

## 4. 兼容与落地建议

## 4.1 存档兼容

## 4.2 代码落地顺序

1. 先改数据模型：每桌独立远征状态与候选快照
2. 再改 UI 流程：取消全局锁、支持窗口随开随关
3. 接入新前置门槛：`max_dice + 1` 双校验
4. 替换 `merge` 算法与稀有度同级晋升
5. 回归测试多桌并行远征与边界情况

## 5. 验收清单

- A 桌远征计时中，关闭窗口后可在 B 桌正常掷骰、结算、开远征
- A/B 两桌可同时处于 `running`
- 某桌远征完成后仅该桌进入 `ready_to_confirm`，不影响其他桌
- 合成开始与确认都遵守 `dice_count >= max_dice + 1`
- 新合成结果满足“打乱配对逐位取最大 + 排序”
- 两骰稀有度相同会正确升一级
